0  How to read this document

The Go port reproduces the Per%Sense amortization logic behind a REST API. Like every Per%Sense screen it is a field-presence solver: the caller fills in what is known, leaves one field blank, and the engine solves for the blank — a payment, a rate, a loan amount, or a term. The forward direction generates a period-by-period repayment schedule; the backward direction inverts that schedule.

The big architectural change since the previous edition: the Go port now ships a faithful structural port of the DOS engine as the default for advanced-option ("fancy") loans. The original piecewise engine in engine.go remains as the entry point and required fallback for everything the port does not yet cover.

Citation convention

References look like engine.go:1264 and resolve under internal/finance/amortization/ unless another path is named (e.g. interest/math.go:134, api/handlers.go:752). Where the Go code cites a Pascal line, that lineage is noted as (AMORTOP.pas:723).

The three functions to anchor on. engine.go:139 Amortize(input) is the package entry — it classifies, then either delegates to the structural port or runs the piecewise path. dosport_entry.go:424 AmortizeDOS(input) is the faithful port — one schedule walk, one Newton solve. engine.go:1264 generateFancySchedule is the piecewise period-by-period heart used as both fallback and backward-solve oracle.

1.2  Pointer fields → field presence

The API request uses pointer types for every field that participates in field-presence dispatch, so a missing JSON key is distinguishable from a supplied zero api/handlers.go:180,278:

Amount *float64 `json:"amount,omitempty"`  // nil => blank => solve the loan amount
Rate   *float64 `json:"rate,omitempty"`    // nil => blank => solve the rate

The handler only stamps InOutInput when the pointer is non-nil api/handlers.go:858,862; a nil pointer leaves the status at Go's zero value StatusEmpty (=0). A blank Payment arrives as a JSON zero rather than a pointer, so it is explicitly downgraded to StatusEmpty api/handlers.go:903. Supplying any advanced option flips input.Fancy = true api/handlers.go:329.

1.3  The status ladder

Each field carries an int8 status forming the dispatch ladder types/constants.go:54–58: InOutBad(−1) < StatusEmpty(0) < InOutOutput(1) < InOutDefault(2) < InOutInput(3). The pervasive guard Status >= InOutDefault means "known" (a default or a user input); Status < InOutDefault means "blank — solve for it." A solved field is written back as InOutOutput (or promoted to InOutInput before a forward run).

The input bundle is LoanInput types.go:145: a Loan, the advanced-option slices, Settings, and the Fancy flag. The core loan types.go:43 pairs each value with a status:

3  First pass & dispatch

FirstPass firstpass.go:43 derives whichever of {first date, last date, #periods} is blank but computable, in three field-presence arms (no iteration): default the first payment date from the loan date + frequency (Amortize.pas:244); derive lastDate = AddNPeriods(firstDate, perYr, n−1) when first + count are known firstpass.go:85; or derive n = NumberOfInstallments(first, last) when first + last are known firstpass.go:102–105. The count is computed on a forced 360 basis firstpass.go:104–105 to match the DOS NumberOfInstallments. LastOK records whether the term is now known — the central flag the dispatch reads.

Amortize engine.go:139 then validates required fields, computes the rate and growth factor, and reaches the central fork:

// engine.go:226 — the gate that picks the engine
if dosPortCanHandle(input, loan, &settings) {
    pin := input
    pin.Loan = loan // post-FirstPass (term/dates derived)
    return AmortizeDOS(pin)
}
// ...else fall through to the piecewise engine below

When the gate declines, the piecewise path forces fancy mode where its simple loop cannot cope (exact-daily, or USA-rule on exact/non-360 — engine.go:235,240), determines the blank field, solves it, and branches to a generator:

4  The two engines & the gate

For an advanced-option loan, the piecewise engine historically solved each segment with a separate sub-loan, which is not identical to the real schedule's tail; stacked options (multi-ARM × balloon × skip × moratorium) could therefore drift by cents. The structural DOS port instead runs one schedule walk and solves every unknown with one Newton over it, exactly as DOS does — so the combinations compose correctly by construction dosport.go:12–17.

What changed. The structural port (dosport*.go, var dosPortEnabled = true at dosport_entry.go:246) is now the default for the validated advanced-option domain, where it matches the real DOS engine to the cent. The original piecewise Amortize is not deleted — it remains the entry point and the required fallback for everything the gate excludes. The port is validated to zero divergence vs the real DOS engine across ~38,000 fuzzed cases (the dosport_fuzz_* sweeps over frequency, first-period, prepaid, basis, given-payment, and the merged option cube).

4.1  dosPortCanHandle — the gate

dosPortCanHandle dosport_entry.go:269 returns true only inside the validated domain. It declines (falling back to piecewise) for any of:

Fallback conditionLine
Port disabled, not fancy, or inBackwardSolve (a piecewise backward solver is using Amortize as its oracle):270

In-advance, Rule-of-78, exact-daily, or daily mode:278
Amount or rate is the unknown (the port solves/uses only the payment):281
REPLACE-mode extras (!PlusRegular with a balloon/prepayment that replaces rather than adds):292
An unbounded / unsupported prepayment shape; AO10 duration is allowed but routed carefully:303

The off-cycle-balloon and all-month-skip exclusions keep the piecewise engine's Rev-10 exact-date balloon draining and its explicit "does not retire" handling in force, where the port would otherwise apply an off-cycle balloon at the next payment.

4.2  The structural DOS port

AmortizeDOS dosport_entry.go:424 mirrors DOS MakeTable: solve the blank payment when one is unknown (estimateAndRefinePayment dosport_entry.go:177), then build the schedule with repayFancyLoan(entire). The engine state lives in a dosEng dosport.go:82 that holds the Pascal module-level "globals" (h^, balloon[], pre[], adj[], mor, targ, skp, the running counters) as one instance per call — pure and testable rather than real package globals. One row in progress is a dpPayment dosport.go:67, a faithful copy of the Pascal paymenttype object (AMORTOP.pas:87).

4.3  ComputeNext, step by step

computeNext(pt, p, usap) dosport.go:213 advances one period in the exact order DOS does (AMORTOP.pas:723–822). The piecewise generateFancySchedule (§5.4) mirrors the same six steps:

  1. Advance the date — step the regular grid one period from base_date, anchored to the first-payment day-of-month.
  2. Skip-month zeroing — if the month is in the skip set, set payamt := 0, else the base payment d.
  3. Find the next extra — findNextExtra returns the soonest balloon and/or co-dated prepayment (summed); balloonpos positions it before (−1: pay only the extra, do not advance the regular grid), coincident (0: add or replace per plus_regular), or after (1). The regular grid advances only when a regular payment occurred (balloonpos >= 0).
  4. Interest interval & accrual — on the 360 basis or non-exact mode the interval snaps to whole calendar months; otherwise it is the exact day-count. Interest accrues on the principal net of the USA bucket (p − usap), rounded to the cent only when the payment is hard.
  5. Moratorium & target — during a moratorium only interest is collected; otherwise, if the period's principal reduction falls short of targ.target, the payment is bumped up to meet it. Target overrides skip-month zeroing (AMORTOP.pas:653).
  6. New balance & USA rule — p := p + interest − payamt; the USA bucket tracks unpaid interest usap := usap + interest − payamt, floored at 0, so principal whose interest is uncovered does not itself earn interest.

5  The piecewise fallback engine

When the gate declines, Amortize runs the original piecewise engine. It is the required fallback (and the oracle the backward solvers drive — §6).

5.1  Rate primitives

All generators share: GrowthPerPeriod engine.go:29 returns the per-period factor f = 1 + rate/RealPerYr(...) (weekly/biweekly use an explicit 7- or 14-day fraction); ComputeTrueRate engine.go:44 derives the continuous rate for daily accrual; firstPeriodProrate engine.go:812 measures the odd first period as a fraction of a normal period; periodYearFraction engine.go:838 is the interior-period analogue; exactDaily engine.go:20 gates true daily accrual on Exact && Basis != 360 (AMORTOP.pas:765).

5.2  The simple loop

generateSimpleSchedule engine.go:875 handles plain (non-fancy) loans. A user-supplied (hard) payment triggers the "Dav Holle provision": per-period interest is rounded to cents engine.go:888–891 (Amortize.pas:1687). The first-period window emits a settlement stub for prepaid loans or prorates the first period's interest. The per-period order of operations:

interest = p*(f-1)              (interior, arrears)        engine.go:1030
interest = p*(f-1)*prorate      (odd first)
daily:   interest = p*(e^(truerate*YearsDif) - 1)
in-advance: interest = (p-pmt)*(f-1)/(2-f)                engine.go:1031
new balance: p = p + interest - pmt                       engine.go:1031

Rule-of-78 (360 basis, non-in-advance) front-loads interest via a constant step r78step = (n·payment − amount)/(0.5·n·(n+1)) decremented each period engine.go:1011–1022 (Amortize.pas:1414). On the final period or when the balance would go non-positive, the residual is folded into the payment engine.go:1100–1109.

5.3  In-advance settlement

generateExactInAdvanceSchedule engine.go:1183 handles DOS's distinct exact×in-advance shape: a row-0 settlement-interest row dated the loan date, a one-period base shift, then n−1 actual-day amortizing rows (AMORTOP.pas:1148–1240). The simple in-advance branch in generateSimpleSchedule engine.go:958–975 emits the equivalent settlement row so the in-advance total is not short by the settlement interest (≈$500 on a 100k/6% monthly loan, the former bug).

5.4  The fancy engine

generateFancySchedule engine.go:1264 is the piecewise period-by-period engine for any advanced-option loan the gate routed here, with a USA-rule exempt-principal bucket usap. veryLast engine.go:1291 sets the schedule terminus (latest of last date, balloons, prepay stops). The main loop mirrors DOS ComputeNext (AMORTOP.pas:723–822):

  1. Off-cycle draining engine.go:1427–1540: any balloon/prepayment dated before the regular date is emitted as its own dated row with partial-period interest from prevDate — applied at its exact date, not folded into the next payment.
  2. Period interest engine.go:1564–1576 on (p−usap), rounded if hard engine.go:1579.
  3. Skip-month zeroing engine.go:1585, then moratorium (interest-only) engine.go:1603.
  4. Coincident extras (balloons + prepayments on this date; PlusRegular adds vs replaces) engine.go:1643–1705.
  5. Target: bump payment to meet the minimum principal reduction — overriding skip-month zeroing engine.go:1725 (AMORTOP.pas:653).
  6. New balance + USA bucket engine.go:1800–1802, then rate adjustments at ARM dates (re-amortize remaining balance, or solve the implied rate from a new payment) engine.go:1846–1908.

6  Backward solving

6.1  Closed-form estimates

Each piecewise solver starts from annuity algebra. SolvePaymentClosedForm backward.go:83: pay = amount·(f−1)/(1−f−n) via Lnn/Exxp, adjusted for first-period proration and in-advance. SolveLoanAmount backward.go:197: amount = (1−f−n)/(f−1)·d + Σ balloon·e−rate·yrs. SolveRate backward.go:298: Newton/secant on the repay residual, first guess payAmt·perYr/amount floored at 0.02. solveNPeriodsFromPayment backward.go:380: n = round(1.4999 + ln(1−p₁(1−ff)/(ff·d))/ln(ff)) (AMORTOP.pas:1660). Balloon and prepayment solves live at backward.go:519,651; the prepayment-duration solve at backward.go:827.

6.2  The schedule-oracle bisection

For fancy loans the closed form is only an estimate; refinement runs against the real forward schedule. Because the engine forces the final payment and stops early, the unforced residual is discontinuous and Newton on it is unstable — so fancybisect.go uses bisection fancybisect.go:10–28. fancyOverUnder fancybisect.go:166 runs Amortize and classifies a trial as under-amortized (+1), over-amortized (−1), or solved (0). fancyBisect fancybisect.go:217 expands the bracket until the sign straddles, then bisects to a tolerance (5e-4 for amount/payment, 1e-7 for rate). solveFancyAmount / Rate / Payment fancybisect.go:275,293,316 wrap it with the unknown pinned (rate capped at 200%).

6.3  The DOS secant port

dosIteratePayment fancybisect.go:114 is a bug-for-bug port of DOS Iterate (AMORTOP.pas:1690), driving repayExactTerminal fancybisect.go:43 (the unforced terminal balance) to zero by finite-difference secant newdelta = delta·p/(final−p), with the DOS divergence brake (count += 5 when the step stops shrinking) and the faithful bestx-after-update ordering. Convergence: |residual| < halfpenny = 0.005 or 20 iterations.

6.4  inBackwardSolve isolation

The piecewise backward solvers (SolveLoanAmount, SolveRate, SolveBalloonAmount, SolvePrepaymentAmount) use Amortize itself as a forward oracle with trial values. To keep those recursive forward runs on the same piecewise engine (and avoid re-entering the port mid-search), each defers beginBackwardSolve()() backward.go:198,299,520,652, which sets the package flag inBackwardSolve dosport_entry.go:259. The gate's very first check dosport_entry.go:270 reads that flag, so any nested Amortize during a backward solve stays piecewise.

The structural heart of the backward path. The closed-form estimate gets the solver close; then the engine is used as its own forward oracle — fancyOverUnder calls Amortize recursively — and bisection/secant drives the real schedule's terminal balance to zero, the same criterion as DOS's Iterate.

7  The advisory layer

Both engines run their result through the same advisory pass so the UI can surface non-fatal warnings. appendResultAdvisories advisories.go:54 inspects the finished AmortResult and the input and appends tiered messages via AmortResult.add advisories.go:145 — e.g. a solved prepayment that retired the loan early (A-W7), a balloon dropped because the payment was computed not input (A-W11), or an ARM whose implied rate is negative (A-W12, emitted once). appendScheduleWarnings advisories.go:22 adds high-rate and rate-status notes. The structural port emits the same advisory codes so the two engines are indistinguishable to the API consumer (dosport_advisory_diff_test.go guards this).

9  Porting notes & gotchas

  • The port is the default; piecewise is the fallback. Do not assume a fancy loan ran the engine.go loop — check the gate engine.go:226. Outside the validated domain (in-advance, R78, exact, off-cycle balloons, AO6/AO7-with-balloon, all-month-skip, amount/rate unknowns, backward-solve oracles) it falls through to piecewise.
  • Exact requires a non-360 basis. exactDaily is true only when both the Exact flag and a non-360 basis are set engine.go:20; on 360, Exact is a no-op (AMORTOP.pas:765).
  • Exact and USA-rule force fancy mode engine.go:235,240 — DOS routes every non-360 loan and every USA-rule loan through the period-by-period engine.
  • Odd-first payment is augmented, not just prorated. The closed-form payment shortcut applies only to plain 360/prepaid/in-arrears/natural-first loans; otherwise the engine iterates. The schedule accrues whole-month interest while the payment is augmented — a deliberate DOS asymmetry, documented inline engine.go:298–340.
  • In-advance blank-payment loans iterate via the annuity-due secant rather than the ordinary-annuity closed form engine.go:563–598.
  • Off-cycle balloons land at their exact date with partial interest, not folded into the next payment engine.go:1461–1540 (AMORTOP.pas:694).
  • Negative implied ARM rate is allowed (overpaying payment), bounded |rate| < 2 — clamping at zero stalled the secant.
  • The fancyBisect-over-Newton choice is intentional — the forced final payment makes the residual a step function; see the rationale block fancybisect.go:10–28.
